# CLAUDE.md — プロジェクト規約

## プロジェクト概要

Flask + DynamoDB で REST API を実装するハンズオンコンテンツの開発プロジェクト。
対象者は Python と Web の基礎を習得済みの社内メンバー（半日・3.5時間）。

---

## ユーザー（Yohei）について

- このプロジェクトの開発者を「Yohei」と呼ぶ
- Web アプリケーション開発は初心者のため、教育的な観点での解説やわかりやすい説明を心がける
- 技術的な判断をする際は、なぜそうするのかの背景も添える

---

## 仕様管理・ナレッジ管理

- **SPEC.md**: ワークショップで作成するものの仕様をまとめたファイル。設計・実装はこのファイルを参照して行う。仕様変更時は SPEC.md を更新する
- **TODOS.md**: タスクの内容を先にこのファイルにまとめてから、タスクを順番に消化する。タスクが一つ終わるごとにこのファイルを更新して進捗状況をまとめる
- **ISSUES.md**: 開発中に発生したエラーやトラブルの内容を記録する。発生状況・原因・対処法をセットで残す
- **KNOWLEDGE.md**: ISSUES.md での対応から得られたナレッジや TIPS をまとめる。今後の開発やハンズオン運営に活かせる知見を蓄積する

## ディレクトリ構成

```
flask_workshop/
├── SPEC.md              # ハンズオン全体仕様

├── ISSUES.md            # エラー・トラブル記録

├── starter/             # 学習者に配布する骨格コード
│   └── app/
├── answer/              # 完成形コード（講師用）
│   └── app/

├── infra/               # AWS CDK（TypeScript）によるインフラ定義

```

## コード・ドキュメントの言語方針

| 対象 | 言語 | 例 |
|---|---|---|
| ドキュメント（.md 等） | 日本語 | 手順書、講師ガイド |
| コード内コメント | 日本語 | `# DynamoDB をスキャンして全書籍を取得する` |
| 変数名・関数名 | 英語 | `get_table()`, `book_id` |
| API エラーメッセージ | 英語 | `"Book not found"` |

基本方針: **システム的なものは英語、人間が説明を受けるためのものは日本語**

## 骨格コードと完成形コードの関係

- **完成形コード（`answer/`）を先に作成**し、そこから実装部分を抜いて **骨格コード（`starter/`）を導出する**
- 骨格コードには日本語コメントでヒントを残す（学習者が何を実装すべきかわかるようにする）
- 骨格コードの関数は `pass` で中身を空にし、コメントで手順を示す

---

## 作業の進め方

- 作業が完了したら、その内容を **TODOS.md に反映**（該当タスクを `[x]` に更新）してから、Yohei に報告する
- 作業ごとに **Git のコミットを作成**する

## Claude Code との協業原則

このプロジェクトの開発を通じて得られた、Claude Code で効果的に開発を進めるための原則。

### ドキュメント駆動

- `SPEC.md`, `TODOS.md`, `CLAUDE.md` がプロジェクトの記憶装置になる
- セッションが切れてもこれらのファイルからコンテキストを復元できるので、常に最新に保つ
- 新しいセッションを開始したら、まずこれらを読んで現状を把握する

### 小さく作って確認する

- 1 タスク 1 コミットで進める。変更の意図が明確になり、問題時の切り戻しも容易
- プレビューやビルドで即座に確認し、短いフィードバックループを回す
- 一度にすべてを完璧にしようとせず、骨格 → 詳細の順で段階的に肉付けする

### 意図を添えて指示する

- 「何を」だけでなく「なぜ」を伝えると、Claude Code は他の箇所にも一貫して意図を適用する
- 例: 「レスポンス構築は毎回書く」＋「API の要だから手で覚える過程を繰り返す」
- 教育コンテンツでは「学習者に何を書かせるか」の判断は人間が行い、ボイラープレートは Claude Code に任せる

### エラーとナレッジの蓄積

- トラブルが起きたら `ISSUES.md` に記録し、得られた知見は `KNOWLEDGE.md` にまとめる
- 同じ問題を繰り返さないための仕組みとして機能する
- セッション間で失われがちな「ハマりポイント」を永続化する
