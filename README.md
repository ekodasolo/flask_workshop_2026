# Flask + DynamoDB ハンズオン — Book Review API

社内メンバー向けの半日（3.5 時間）ハンズオンコンテンツです。
Flask で REST API を実装し、DynamoDB にデータを保存、Docker コンテナにして AWS (ECS Fargate) にデプロイするまでを体験します。

## 作るもの

書籍とレビューを管理する REST API（Book Review API）。

- 書籍の CRUD（一覧・登録・詳細・更新・削除）
- レビューの投稿・一覧取得
- DynamoDB シングルテーブル設計（PK/SK パターン）

## 対象者

- Python の基本文法を理解している
- Web の基礎（HTTP、JSON、REST）を知っている
- AWS を使ったことがある（コンソール操作程度で OK）

## 開発手法 — Claude Code によるスペック駆動開発

このプロジェクト自体を **Claude Code（Anthropic の AI コーディングエージェント）** と人間の協業で開発しています。採用しているのは「スペック駆動開発」の簡易版です。

### スペック駆動開発とは

ソフトウェアの**仕様書（Spec）を開発の起点かつ信頼できる唯一の情報源（Single Source of Truth）として扱う**開発手法です。コードを書く前にまず仕様を定め、実装・テスト・ドキュメントのすべてを仕様から導出します。

AI コーディングエージェントとの協業において特に有効で、仕様書があることで：
- セッションが切れても AI がコンテキストを復元できる
- 人間と AI の間で「何を作るか」の認識がずれにくい
- 仕様変更の影響範囲を把握しやすい

### 本格的なスペック駆動開発のプロセス

```
1. 仕様策定    → Spec を書く（要件・API 定義・データモデル・制約）
2. タスク分解  → Spec からタスクを導出し、依存関係を整理する
3. 実装       → タスクを 1 つずつ消化し、Spec に沿ってコードを書く
4. 検証       → Spec に対してテスト・レビューを行う
5. フィードバック → 発見事項を Spec にフィードバックし、仕様を更新する
```

各フェーズで AI と人間の役割を明確に分けるのがポイントです。

### このプロジェクトでの実践（簡易版）

本格的な Spec 駆動ではテスト自動化やCI/CDとの連携も含みますが、このプロジェクトでは以下の簡易構成で実践しています。

#### 管理ファイル体系

| ファイル | 役割 | スペック駆動での位置づけ |
|---------|------|----------------------|
| `SPEC.md` | ハンズオン全体の仕様 | **Spec 本体** — 何を作るかの定義 |
| `TODOS.md` | タスク管理と進捗 | **タスク分解** — Spec から導出した作業リスト |
| `CLAUDE.md` | プロジェクト規約 | **開発ルール** — AI への指示と制約 |
| `ISSUES.md` | トラブル記録 | **フィードバック** — 実装中の発見事項 |
| `KNOWLEDGE.md` | ナレッジ蓄積 | **学習成果** — 再利用可能な知見 |

#### 人間と AI の役割分担

| 判断の種類 | 担当 | 例 |
|-----------|------|-----|
| 何を作るか（仕様） | 人間 | API 設計、学習目標、対象者レベル |
| なぜそうするか（意図） | 人間 | 「レスポンス構築は学習者に書かせる」 |
| どう作るか（実装） | AI | コード生成、ドキュメント作成、CSS 調整 |
| 正しいか（レビュー） | 人間 | 方針の確認、教育的観点でのフィードバック |
| 記録する（蓄積） | AI + 人間 | ISSUES/KNOWLEDGE の記録、TODOS の更新 |

#### 開発サイクル

```
人間: SPEC.md で仕様を決める
  ↓
人間: TODOS.md にタスクを書き出す
  ↓
AI:   タスクを 1 つ実装 → コミット → TODOS.md を更新
  ↓
人間: 確認・フィードバック（意図を添える）
  ↓
AI:   フィードバックを反映 → 次のタスクへ
  ↓
(問題発生時) AI: ISSUES.md に記録 → KNOWLEDGE.md にナレッジ化
```

### この手法のメリット

- **コンテキストの永続化**: AI のセッションが切れても、管理ファイル群を読めば続きから作業できる
- **品質の一貫性**: CLAUDE.md の規約に従うことで、セッションをまたいでもコードスタイルや方針がぶれない
- **判断の分離**: 仕様判断は人間、実装は AI という明確な役割分担で、双方の強みを活かせる
- **知識の蓄積**: ISSUES.md → KNOWLEDGE.md の流れで、プロジェクト固有のノウハウが失われない
